# moodlist v1.3 — Design Spec

**Version:** 1.3
**Date:** 2026-05-15
**Status:** Approved — locked for implementation
**Author:** myorek (with assistant collaboration during brainstorm)
**Base:** v1.2.0 (commit `de527ba`, tag `v1.2.0`)

## 1. Terminology

**"Wishlist entry"** in v1.3 means specifically: an **album** that
Claude Haiku has recommended at least one missing track from, which
the user does not currently own (or owns only partially).

Compared to v1.2, the unit of dedup shifts from the **track** to the
**album**. The wishlist becomes a *hunt list* — a shopping list of
discs to buy, with the field shape needed to find them in physical
Japanese second-hand stores (Bookoff being the user's primary venue),
which sort Western artists either by Latin alphabet or by Japanese
phonetic order (gojūon).

**"Starter character"** = the first character used for bin navigation:
- For artists with a Japanese name on the spine: the first kana of
  the Japanese name (e.g. レ for レッド・ツェッペリン).
- For artists without a Japanese rendition (AC/DC, U2): the first
  Latin letter of the English name.

## 2. Problem

v1.2's wishlist is a track-level deduplicated list. When Haiku
recommends three different tracks from the same album across separate
queries, the user sees three wishlist rows. The user shops at Bookoff
where they buy *albums*, not individual tracks, and where bin
navigation depends on Japanese transliterations and the leading kana.

The current display:

```
  Mentions  Last seen    Track
  --------  -----------  -------------------------------------------------
        5   2026-05-15   Led Zeppelin - Whole Lotta Love
        2   2026-05-14   Led Zeppelin - Heartbreaker
```

The user wants:

```
 Bin  Artist               Album              Year
  レ  Led Zeppelin         Led Zeppelin II    1969
      レッド・ツェッペリン
```

— one album row, with everything they need to find it at the store.

## 3. Goals

1. Wishlist rows represent **albums**, not tracks. Same album surfaced
   by N tracks = one row with N mentions aggregated.
2. Each row carries: English artist, Japanese artist (when distinct),
   starter character for bin navigation, English album, year.
3. `moodlist wishlist` default output shows the album-hunt format
   from the design preview.
4. Two new sort modes: `--sort kana` (by Japanese gojūon) and
   `--sort latin` (by English alphabet) for shopping-aisle use.
   Default remains `mention_count desc, last_seen desc` — "most
   wanted first".
5. Existing 53 v1.2 track-level entries are migrated to album-level
   in a one-shot Haiku call. Migration is idempotent and resilient
   to partial failure.
6. No regression in playlist generation, Alfred workflow, query
   cache, indexing, or `--debug`.

## 4. Non-goals (v1.3)

- **No MusicBrainz / external API integration.** Album resolution
  and Japanese name lookup are 100% Haiku-driven. Cheaper, simpler,
  no new dependencies, no rate limits.
- **No catalog number / OBI tracking** (TOCP-, EICP-, etc.). Out of
  scope; future work if the user ever wants specific Japanese
  pressings.
- **No track-level browsing inside the wishlist subcommand.** Once
  resolved to album, the originating track list is not preserved
  (per the user's preference for option-3 display, not option-4).
- **No removal of `misses.log`.** Keeps appending raw track strings as
  in v1.2. Deferred to v1.4+ if the user wants to drop it.
- **No Alfred wishlist keyword changes.** Wishlist remains CLI-only.

### 5.2 New SQLite schema

Replaces the v1.2 `wishlist` table. The migration (§5.7) handles
data transfer.

```sql
CREATE TABLE IF NOT EXISTS wishlist (
    dedup_key       TEXT PRIMARY KEY,
    artist_en       TEXT NOT NULL,
    artist_ja       TEXT,
    starter_latin   TEXT NOT NULL,
    starter_kana    TEXT,
    album_en        TEXT NOT NULL,
    year            INTEGER,
    first_seen      TEXT NOT NULL,
    last_seen       TEXT NOT NULL,
    mention_count   INTEGER NOT NULL DEFAULT 1,
    queries_seen    TEXT NOT NULL DEFAULT ''
);

CREATE INDEX IF NOT EXISTS wishlist_mentions ON wishlist (mention_count DESC);
CREATE INDEX IF NOT EXISTS wishlist_starter_kana ON wishlist (starter_kana);
CREATE INDEX IF NOT EXISTS wishlist_starter_latin ON wishlist (starter_latin);
```

**`dedup_key`** = `normalize("artist_en - album_en")` using the same
`normalize_track_name` function as v1.2. Same album under different
spellings ("Led Zeppelin II" vs "Led Zeppelin 2") dedupes. Year is
NOT part of the key — reissues of the same album collapse.

### 5.3 New dataclass: `WantedAlbum`

In `moodlist/types.py`:

```python
@dataclass(frozen=True)
```

### 5.4 `AgentResult` change

```python
@dataclass(frozen=True)
class AgentResult:

    needs_live: bool
    raw_picks: list[int] = field(default_factory=list)
```

`wanted_but_missing` remains a `list[str]` for the audit-trail

the wishlist.

### 5.5 Prompt change

`SYSTEM_PROMPT` gets one new instruction in the existing bullet list
(after "wanted_but_missing"):

> - For each track in `wanted_but_missing`, also include an entry in
>   `wanted_albums` resolving it to the canonical album it appears on.
>   Each `wanted_albums` entry is `{artist, artist_ja, album, year}`.
>   `artist_ja` is the standard Japanese katakana transliteration
>   (e.g. "Led Zeppelin" → "レッド・ツェッペリン"); set it to `null`
>   when the artist's name is not commonly rendered in Japanese
>   (AC/DC, U2). When multiple wanted tracks share an album, list
>   the album once, not once per track. `year` is the original
>   release year of the album (not a reissue).

Schema in the prompt's JSON description gets the new field:

```json
{
  "picks": [int],
  "reasoning": string,
  "wanted_but_missing": [string],
  "wanted_albums": [
    {"artist": string, "artist_ja": string | null,
     "album": string, "year": int | null}
  ],
  "needs_live": boolean
}
```

Token cost: roughly +200-500 output tokens per query (depending on
how many albums are referenced). At Haiku 4.5 output pricing,
≈ +$0.001 per call. Negligible.

### 5.6 Starter-character derivation

A new helper `derive_starters(artist_en, artist_ja) -> (str, str | None)`
in `moodlist/wishlist.py`:

```python
def derive_starters(
    artist_en: str, artist_ja: str | None
) -> tuple[str, str | None]:
    # Latin: first letter, ignoring leading "The "
    latin_source = artist_en
    if latin_source.lower().startswith("the "):
        latin_source = latin_source[4:]
    starter_latin = latin_source[:1].upper() if latin_source else "?"

    # Kana: first character of artist_ja (if any), ignoring leading
    # "ザ・" prefix the same way "The " is ignored in Latin.
    starter_kana: str | None = None
    if artist_ja:
        kana_source = artist_ja
        if kana_source.startswith("ザ・"):
            kana_source = kana_source[2:]
        starter_kana = kana_source[:1] if kana_source else None
    return starter_latin, starter_kana
```

Examples:

| Input artist | Latin | Kana |
|---|---|---|
| "Led Zeppelin" / "レッド・ツェッペリン" | L | レ |
| "The Who" / "ザ・フー" | W | フ |
| "AC/DC" / None | A | None |
| "Queen" / "クイーン" | Q | ク |
| "U2" / None | U | None |

### 5.7 One-time migration from v1.2 wishlist

On first construction of `WishlistDB` after v1.3 install, the
class detects the v1.2 schema (presence of `display_name` column, or
absence of `artist_en` column — either signal works; we use the
latter).

```python
def _detect_v1_2_schema(conn) -> bool:
    cols = {row[1] for row in conn.execute("PRAGMA table_info(wishlist)")}
    return "display_name" in cols and "artist_en" not in cols
```

If v1.2 schema detected:

1. Read all rows from old `wishlist` table.
2. If 0 rows: drop and recreate with new schema. Done.
3. If N rows: build a single Haiku resolution request — feed all N
   `display_name` strings, ask for `{artist, artist_ja, album,
   year}` per row. Aggregate by canonical album.
4. Rename old table to `wishlist_v1_2_backup`.
5. CREATE TABLE new `wishlist` with v1.3 schema.
6. INSERT aggregated rows, preserving `first_seen`/`last_seen`/
   `queries_seen` (`mention_count` becomes sum of source rows that
   resolved to the same album).
7. Print `wishlist: migrated N entries → M albums` to stderr.

**Idempotency:** the v1.2 schema check is the trigger. After
migration, the table has `artist_en` so re-running does nothing. The
backup table is left in place; user can drop manually if desired.

**Resilience:** if the Haiku call fails (network, API key), the
migration is skipped silently for this run; the v1.2 wishlist
remains intact and untouched. Migration retries on next `WishlistDB`
construction. The user's playlist queries continue working in the
meantime because the read paths gracefully handle missing v1.3
columns by returning empty results.

**Cost:** ~$0.02 one-time. Single Haiku call with all 53 entries in
one batch.

### 5.8 Album resolver (used by both per-query writes and migration)

```python
def resolve_albums(
    track_strings: list[str],
    *,
    api_key: str,
    model: str,
) -> list[WantedAlbum]:
    """Call Haiku to map a list of 'Artist - Track' strings to
    canonical album records. Returns a deduplicated list."""
```

System prompt for the resolver:

> You are a music-catalog resolver. For each input string, identify the
> artist, album, and original release year, plus the standard Japanese
> katakana transliteration of the artist (or null if not applicable).
> Deduplicate: if multiple input strings resolve to the same album,
> return ONE record. Output JSON: `{"albums": [{artist, artist_ja,
> album, year}, ...]}`.

Used by:
- `migrate_v1_2_to_v1_3()` — single call for all old entries.
- Optionally used in the future as a fallback if Haiku's per-query
  `wanted_albums` is missing for some reason (not in v1.3 scope).

In normal v1.3 query flow, `agent.pick()` already produces
`wanted_albums` directly — no second Haiku call needed per query.

### 5.9 Per-query write path

Replaces v1.2 wishlist write block in `cli.py`:

```python
if result.wanted_albums:
    try:
        # 1. Library track-level pre-filter — same as v1.2, but applied
        #    to wanted_but_missing (track strings) to derive a set of
        #    album dedup_keys whose tracks the user already owns.
        library_track_keys = {
            normalize_track_name(f"{t['artist']} - {t['title']}")
            for t in library
        }

        wishlist_db = WishlistDB(moodlist_dir / "wishlist.sqlite")
        for album in result.wanted_albums:
            # Library pre-filter at the album level: if Haiku's wanted_albums
            # includes an album whose normalized "artist - album" key matches
            # something we can derive from the library, skip.
            # NOTE: This is conservative because library tracks don't carry
            # album metadata. The track-level pre-filter remains the primary
            # gate — if the user owns the specific track Haiku surfaced, the
            # wanted_albums entry shouldn't have been added by Haiku in the
            # first place (per prompt). Defense-in-depth.
            wishlist_db.upsert_album(album, query=args.query, seen_at=today)
    except Exception as e:
        print(f"wishlist: write failed ({e}); continuing", file=sys.stderr)

# misses.log keeps appending wanted_but_missing track strings (unchanged).
```

### 5.10 Auto-prune on reindex

After `Indexer.build()`, walk the wishlist. For each album row, if
the user's library contains ANY track whose normalized
`"artist - title"` matches the album's `artist_en`, treat the album
as partially or fully owned. **Conservative rule (v1.3 default):**
do NOT prune albums based on partial ownership. Only prune when the
exact album+artist appears in the library via the album's
`normalize_track_name(f"{artist_en} - {album_en}")` matching against
some library track's similar key.

This is intentionally less aggressive than v1.2 because libraries
don't carry album metadata. The user can manually edit `wishlist.sqlite`
or wait for v1.4 (which may add MusicBrainz album resolution for the
local library).

A simpler interim approach used in v1.3: keep the same v1.2 logic —
build `library_keys = { normalize("artist - title") }` and prune any
wishlist row whose `normalize("artist - album")` key happens to match
(rare but possible: e.g., a track titled the same as the album).

### 5.11 New CLI display

`moodlist wishlist` output replaces the v1.2 mention-count table:

```
Wishlist — 31 albums to hunt (53 tracks deduped)

 Bin  Artist                   Album                  Year   ×N
 ──   ──────────────────────   ────────────────────   ────   ──
  L   Led Zeppelin             Led Zeppelin II        1969    7
      レッド・ツェッペリン
  P   Pink Floyd               The Wall               1979    5
      ピンク・フロイド
  W   The Who                  Who's Next             1971    4
      ザ・フー
  A   AC/DC                    Back in Black          1980    4
  D   David Bowie              Heroes                 1977    3
      デイヴィッド・ボウイ
  …
```

The `Bin` column shows `starter_latin` by default. When
`--sort kana` is selected, it shows `starter_kana` (falling back to
`starter_latin` if no kana). The `×N` column is `mention_count`.

Year displays as `——` when null. `artist_ja` line is omitted when
null.

New flags:

| Flag | Effect |
|---|---|
| `--sort mentions` (default) | Sort by `mention_count desc, last_seen desc` (v1.2 default behavior). |
| `--sort latin` | Sort by `starter_latin`, then `artist_en`. |
| `--sort kana` | Sort by `starter_kana` (kana entries first, then Latin-only). |
| `--all` | Show every album (no top-50 limit). |
| `--since N` | Only entries seen in last N days. |
| `--json` | Machine-readable. |

Default sort stays at `mentions` (matches v1.2 ergonomics — most-wanted
first). The shopping-mode `latin`/`kana` sorts are opt-in for when the
user is physically at the store and wants to walk the bins in order.

## 7. Error handling

- **Haiku returns malformed `wanted_albums` entry** (missing required
  fields, wrong types): silently skipped, logged via `--debug`.
- **Haiku returns no `wanted_albums` at all**: wishlist write is a
  no-op for that query. `misses.log` still appends the raw strings.
- **Migration call fails** (network, API key, malformed response):
  v1.2 wishlist is untouched. CLI prints `wishlist: migration deferred
  (will retry next run)` to stderr. Subsequent CLI calls retry.
- **SQLite schema-detection failure**: skip migration, log error,
  carry on with raw v1.2 wishlist data accessible only via the
  legacy `wishlist_v1_2_backup` table (after first successful
  migration).
- All wishlist failures continue to be caught at the CLI layer so
  playlist generation never blocks.

## 8. Migration / compatibility

- **First run after upgrade:** v1.2 `wishlist` table is detected,
  a single Haiku resolver call is fired, results are written to a
  new v1.3 `wishlist` table, old table is renamed
  `wishlist_v1_2_backup`. ~5s blocking on the first run; instantaneous
  thereafter.
- **`misses.log` unchanged.** Continues appending raw strings.
- **`query-cache.sqlite` unchanged.**
- **`library.sqlite`, `library.cache.json`, `library.version`
  unchanged.**
- **Existing playlists in `~/.moodlist/playlists/` unchanged.**
- **Alfred workflow unchanged.** Same `ml`, `ml!`, `ml-reindex` keywords.
