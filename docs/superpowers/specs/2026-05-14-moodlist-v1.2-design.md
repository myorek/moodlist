**Version:** 1.2
**Date:** 2026-05-14
**Status:** Approved вАФ locked for implementation
**Author:** myorek (with assistant collaboration during brainstorm)
**Base:** v1.1.1 (commit `de0e787`, tag `v1.1.1`)

## 1. Terminology

**"Missing track" / "wishlist entry"** means specifically: a track
that **Claude Haiku recommended** as a canonical match for the user's
query (via the `wanted_but_missing` field in the model's JSON
response), but which **the user does not currently own** in their
local FLAC library.

It is not "a track from a chart". It is not "a track the user wants".
It is a track *Haiku thinks fits the query but isn't in the library*
вАФ effectively the model's "if you owned this, I'd have picked it"
list. The user can then choose to buy/acquire it and add it to
`~/Music/`.

## 2. Problem

In v1.1, `misses.log` accumulates these Haiku-recommended-but-not-owned
suggestions as a flat append-only text log. With 69 lines of
suggestions in the user's log, only 55 are unique вАФ duplicates
accumulate across repeat queries. Worse, the log includes things the
user already owns (Haiku doesn't always know what's in the library
after the fact, so it sometimes recommends tracks the user already has
under a different spelling).

The user wants a usable shopping list:

- **Deduplicated**, so "Black Sabbath - Paranoid" appears once no
  matter how many metal queries surface it.
- **Filtered against the local library**, so things the user already
  owns under any spelling don't clutter the list.
- **Auto-cleaned** after `moodlist --reindex` вАФ if you buy a FLAC, drop
  it in `~/Music`, and rescan, that entry disappears from the
  wishlist.
- **Printable** via a new `moodlist wishlist` CLI command.

## 3. Goals

1. `wanted_but_missing` suggestions from every cache-miss flow into a
   deduplicated SQLite wishlist.
2. Suggestions that already match a library track (normalized) never
   reach the wishlist in the first place.
3. `moodlist --reindex` auto-prunes wishlist entries that are now in
   the library.
4. `moodlist wishlist` prints a readable table of unique missing
   tracks, sorted by mention count then recency.
5. Old `misses.log` continues to append as an audit trail; v1.2 reads
   it once to migrate prior history.
6. No new third-party dependencies.

## 4. Non-goals (v1.2)

- **No "acquired" marking command** вАФ acquisition is implicit via
  reindex. If you bought a track and didn't reindex, it's still on
  the list. That's the intended UX.
- **No fuzzy matching beyond normalize-exact** вАФ "Paranoid" alone
  won't dedupe against "Black Sabbath - Paranoid". A v1.3 concern if
  it matters in practice.
- **No Alfred `ml-wishlist` keyword** вАФ CLI only. Add later if useful.
- **No purchase-link integration** (Bandcamp, Qobuz, Beatport).
- **No Last.fm / live charts** (still v2-themed).
- **No clipboard side-effects** вАФ `moodlist wishlist` is pure stdout.

### 5.2 SQLite schema

```sql
CREATE TABLE IF NOT EXISTS wishlist (
    dedup_key      TEXT PRIMARY KEY,    -- normalize(display_name)
    display_name   TEXT NOT NULL,       -- original Haiku text, first version observed
    first_seen     TEXT NOT NULL,       -- ISO 8601 (date only is fine)
    last_seen      TEXT NOT NULL,
    mention_count  INTEGER NOT NULL DEFAULT 1,
    queries_seen   TEXT NOT NULL DEFAULT ''   -- newline-separated set
);
CREATE INDEX IF NOT EXISTS wishlist_last_seen ON wishlist (last_seen DESC);
CREATE INDEX IF NOT EXISTS wishlist_mentions  ON wishlist (mention_count DESC);
```

### 5.3 Normalization (`normalize_track_name`)

Same algorithm as `cache.normalize_query` from v1.0, applied to track
names:

1. Unicode NFKC normalization.
2. Lowercase.
3. Strip leading/trailing whitespace.
4. Collapse all whitespace runs to a single space.
5. Strip ASCII punctuation: `.,;:"'вАЩ`!?()[]{}вАФвАУ-/\\|`. (Replace with
   space; then re-collapse whitespace.)
6. Strip "feat. X", "ft. X", "(live)", "(remastered)", "(demo)",
   "(alternate version)", "(extended mix)" parenthetical tails before
   step 5.

Example transformations:

| Input | Output |
|---|---|
| `"Black Sabbath - Paranoid"` | `"black sabbath paranoid"` |
| `"Black Sabbath: Paranoid"` | `"black sabbath paranoid"` |
| `"black sabbath вАУ paranoid"` | `"black sabbath paranoid"` |
| `"Somebody to Love (alternate version)"` | `"somebody to love"` |
| `"Metallica вАУ For Whom the Bell Tolls (1986)"` | `"metallica for whom the bell tolls 1986"` |

A `normalize_track_name` function lives in `moodlist/wishlist.py` and
is the **only** normalization path for both write-time dedup and
reindex-time auto-prune.

### 5.4 WishlistDB public interface

```python
class WishlistDB:
    def __init__(self, db_path: Path) -> None: ...

    def upsert(
        self,
        display_name: str,
        query: str,
        seen_at: str,            # ISO date
    ) -> bool:
        """Insert or update one entry. Returns True if newly inserted,
        False if updated. Idempotent on (dedup_key)."""

    def remove_matching(self, library_keys: set[str]) -> int:
        """Delete every wishlist entry whose dedup_key is in
        library_keys. Returns the number of rows deleted."""

    def list(
        self,
        *,
        limit: int | None = 50,
        since: _dt.date | None = None,
    ) -> list[WishlistEntry]:
        """Return entries ordered by mention_count desc, then
        last_seen desc. `limit=None` means all."""

    def count(self) -> int: ...
```

`WishlistEntry` is a frozen dataclass with the row's six columns
(plus `dedup_key` for completeness).

### 5.5 Write path (cache-miss flow in `cli.py`)

Today the cache-miss flow writes to `misses.log`. v1.2 adds the
wishlist write next to it, with a pre-filter:

```python
# Build the set of normalized keys already in the library
library_keys = {
    normalize_track_name(f"{t['artist']} - {t['title']}")
    for t in library
}

if result.wanted_but_missing:
    # 1. Existing misses.log behavior вАФ unchanged
    misses = moodlist_dir / "misses.log"
    with misses.open("a") as f:
        for m in result.wanted_but_missing:
            f.write(f"{today}\t{args.query}\t{m}\n")

    # 2. NEW: wishlist upsert with library pre-filter
    wishlist_db = WishlistDB(moodlist_dir / "wishlist.sqlite")
    for m in result.wanted_but_missing:
        key = normalize_track_name(m)
        if key in library_keys:
            continue                # already owned, skip
        wishlist_db.upsert(display_name=m, query=args.query, seen_at=today)
```

The `library_keys` set is built once per CLI run (not per entry).

### 5.6 Reindex path (`moodlist --reindex`)

After `Indexer.build()` completes successfully, auto-prune:

```python
if args.reindex or ix.is_stale():
    ix.build()
    # NEW v1.2: prune wishlist of newly-owned tracks
    library = ix.load_compact()
    library_keys = {
        normalize_track_name(f"{t['artist']} - {t['title']}")
        for t in library
    }
    wishlist_db = WishlistDB(moodlist_dir / "wishlist.sqlite")
    removed = wishlist_db.remove_matching(library_keys)
    if removed:
        print(f"wishlist: removed {removed} entries now in library",
              file=sys.stderr)
```

Idempotent вАФ re-running reindex with no new tracks removes nothing.

### 5.7 Read path вАФ `moodlist wishlist` subcommand

Implemented as argparse **subparsers**. Today `moodlist` has a single
flat parser; v1.2 introduces:

```
moodlist [global-flags] <subcommand> [subcommand-flags]
```

with two subcommands:

- `moodlist [...]` (no subcommand) вАФ runs a query, same as today.
  This requires `subparsers(required=False)` and a default behavior
  when no subcommand is given. **Crucially, this must not break any
  v1.0/v1.1 CLI invocation.** A pre-flight check: if the first
  non-flag arg is `wishlist`, dispatch to the wishlist branch;
  otherwise everything else flows through the v1.1 main() body.

- `moodlist wishlist [--all] [--since N] [--json]` вАФ print the wishlist.

To keep backwards compatibility simple, we'll implement subcommand
dispatch manually rather than via argparse subparsers (which would
require restructuring the whole arg block). Logic at the top of
`main()`:

```python
if argv is None:
    argv = sys.argv[1:]
if argv and argv[0] == "wishlist":
    return _wishlist_command(argv[1:])
# else fall through to existing query-running logic
```

`_wishlist_command(argv)` has its own small parser:

```python
def _wishlist_command(argv: list[str]) -> int:
    parser = argparse.ArgumentParser(prog="moodlist wishlist")
    parser.add_argument("--all", action="store_true",
                        help="show every entry, not just top 50")
    parser.add_argument("--since", type=int, metavar="DAYS",
                        help="only entries seen in the last N days")
    parser.add_argument("--json", action="store_true",
                        help="emit machine-readable JSON")
    args = parser.parse_args(argv)
    ...
```

**Output formats:**

Plain mode (default):

```
Wishlist вАФ 55 unique tracks
(showing top 50; run `moodlist wishlist --all` for full list)

  Mentions  Last seen    Track
  --------  -----------  -------------------------------------------------
        3   2026-05-14   Black Sabbath - Paranoid
        2   2026-05-14   Iron Maiden - Run to the Hills

        1   2026-05-14   Pink Floyd - Comfortably Numb
        1   2026-05-14   Aphex Twin - Windowlicker

```

If `count == 0`:

```

  moodlist "top 80s metal"
```

`--json` mode prints one JSON array of objects to stdout:

```json
[
  {
    "display_name": "Black Sabbath - Paranoid",
    "first_seen": "2026-05-14",
    "last_seen": "2026-05-14",
    "mention_count": 3,
    "queries_seen": ["top 80s metal", "best metal", "top rock classics"]
  },
  ...
]
```

### 5.8 One-time migration from `misses.log`

`WishlistDB.__init__` performs a **first-run migration**:

1. If `wishlist.sqlite` doesn't exist (or its `wishlist` table is
   empty) AND `misses.log` exists, run the migration.
2. Read `misses.log` line by line, parse `<date>\t<query>\t<display>`.
3. For each row, call `upsert(display, query, date)` вАФ natural dedup
   via the `dedup_key` PK.
4. Library pre-filter does NOT apply to the migration (the user might
   have owned a track *and* had it on the log historically; we keep
   that data, and the next `moodlist --reindex` will prune it).

The migration is idempotent: once `wishlist` table has any row, the
migration won't run again. Users can force re-migration by deleting
`wishlist.sqlite`.

## 7. Error handling

- `WishlistDB` operations use `try/finally` for sqlite connection
  hygiene, mirroring the v1.0 cache module.
- If `misses.log` is malformed (a line with fewer than 3 tab-separated
  fields), the migration logs to stderr and skips the line вАФ never
  crashes.
- If `wishlist.sqlite` is corrupted, `moodlist wishlist` prints an
  error to stderr suggesting deletion; running queries continues to
  work (we don't block playlist generation on wishlist failures).
- Wishlist write failures during cache-miss flow are caught and
  printed to stderr but do NOT propagate вАФ a wishlist hiccup must
  never break playlist generation.

## 8. Migration / compatibility

- **Existing users:** First `moodlist`-anything run after upgrading
  creates `wishlist.sqlite` and migrates `misses.log`. No user
  action required.
- **`misses.log` keeps working** вАФ wishlist write happens alongside,
  not instead. Audit trail is preserved.
- **No schema changes to existing dbs.** `library.sqlite` and
  `query-cache.sqlite` are untouched.
