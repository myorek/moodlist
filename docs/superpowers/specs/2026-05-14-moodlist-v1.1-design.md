# moodlist v1.1 — Design Spec

**Version:** 1.1
**Date:** 2026-05-14
**Status:** Approved — locked for implementation
**Author:** myorek (with assistant collaboration during brainstorm)
**Base:** v1.0.0 (commit `091d666`, tag `v1.0.0`)

## 1. Problem

Three small issues surfaced during real use of v1.0:

1. **Count ignored.** `ml top 10 rock` returns 20 tracks because the
   prompt always tells Haiku `Desired count: 20` regardless of what the
   user's query says.
2. **No diagnostic mode.** When picks feel "off" — wrong era, too few
   metal, surprising omissions — there's no way to see what Haiku saw,
   what it returned, or why those tracks won.
3. **No visibility into the prompt or library context.** A user who
   suspects "we're sending too few tracks" cannot verify it without
   reading the source code.

## 2. Goals

1. `ml top 10 rock` returns 10 tracks; `ml a dozen rock songs` returns
   12; `ml 5 Queen songs` returns 5. `--count N` keeps working as an
   explicit override.
2. `moodlist --debug "<query>"` prints structured diagnostic sections
   to **stderr**, so stdout stays clean for Alfred / piping.
3. In `--debug` mode only, Haiku additionally returns a one-line
   reason per pick. Outside debug mode this costs nothing.

## 3. Non-goals (v1.1)

- Live charts via Last.fm (deferred to v2).
- Alfred workflow changes (any improvement here is a v1.2 concern).
- CI, GitHub Actions, PyPI publishing.
- New CLI subcommands (`moodlist explain`, etc.).
- Cache schema changes — `query-cache.sqlite` keeps its v1 schema.
- Per-track reasoning visible in normal CLI / Alfred output.
- Regex-based count parsing — Haiku handles all natural-language count
  forms via prompt instruction.

### 4.2 Behavior surface

#### 4.2.1 Count from query

**Today's prompt** (verbatim from `agent.py`):

```
Query: "<query>"
Date: <date>
Desired count: <desired_count>
```

**v1.1 prompt** (the user message):

```
Query: "<query>"
Date: <date>
Default count: <desired_count>
```

**v1.1 system prompt addition** (one bullet added to existing list):

> If the user's query specifies a number of tracks (e.g. "top 10",
> "5 songs", "a dozen tracks"), honor that number. Otherwise return
> the default count given below.

The agent's `desired_count` parameter remains a hint, never a hard
constraint. `--count N` overrides what gets sent as "Default count" in
the prompt.

#### 4.2.2 `--debug` flag

```bash
moodlist --debug "top 10 rock"
```

Behavior:
- Runs the **exact same** pipeline as without `--debug` (config load,
  reindex check, cache lookup, agent call, playlist write, foobar
  launch).
- In addition, emits diagnostic sections to **stderr**.
- Stdout remains exactly what it would be without `--debug` (a
  playlist path under `--dry-run`, nothing otherwise).
- `--debug` works with `--dry-run`, `--fresh`, `--count`, `--open`,
  `--alfred-json`. With `--alfred-json` the debug output still goes to
  stderr so it doesn't pollute the JSON.

#### 4.2.3 `AgentResult` new fields

```python
@dataclass(frozen=True)
class AgentResult:
    picks: list[int]                       # validated against library
    reasoning: str                         # one-line overall summary
    wanted_but_missing: list[str]          # for misses.log
    needs_live: bool
    raw_picks: list[int]                   # NEW: pre-validation picks from Haiku
    pick_reasons: dict[int, str]           # NEW: per-track reasons; empty when debug=False
```

Two new fields with safe defaults (`raw_picks=[]` and
`pick_reasons={}`). Existing call sites continue to compile.

#### 4.2.4 `agent.pick(debug=False)`

Signature:

```python
def pick(
    *,
    query: str,
    library: list[dict],
    date_iso: str,
    api_key: str,
    model: str,
    temperature: float,
    desired_count: int = 20,
    debug: bool = False,    # NEW
) -> AgentResult:
```

When `debug=True`:
- `SYSTEM_PROMPT + DEBUG_SUFFIX` is sent as the system prompt.
- The response schema instructs Haiku to add a `pick_reasons` object
  to its JSON.
- `_extract_json` parses the extra field; `AgentResult.pick_reasons`
  is populated.

When `debug=False`:
- The system prompt is plain `SYSTEM_PROMPT`.
- Response schema is unchanged.
- `AgentResult.pick_reasons` is the empty dict.

#### 4.2.5 `DEBUG_SUFFIX`

```python
DEBUG_SUFFIX = """

DIAGNOSTIC MODE — include per-track reasons:

Additionally, return a `pick_reasons` field in your JSON: an object
keyed by each picked id (as a string), mapped to a one-sentence
explanation of why that specific track fits the query. Keep each
reason under 100 characters. Example:

  "pick_reasons": {
    "42": "Defining 1979 hard-rock single, anchors any rock canon",
    "17": "Most accessible Metallica track; 80s metal bedrock"
  }

The overall `reasoning` field stays as-is (one-line summary).
"""
```

### 4.3 Cost model

| Mode | Input tokens | Output tokens | Latency | $ per call |
| --- | --- | --- | --- | --- |
| Normal (cold) | ~13k | ~300 | ~1.5s | ~$0.003 |
| Normal (prompt-cache hit) | ~1.3k effective | ~300 | ~0.8s | ~$0.0008 |
| Query-cache hit | 0 | 0 | ~50ms | $0 |
| **Debug (cold)** | **~13k + ~150** | **~900-1300** | **~3-4s** | **~$0.008** |
| **Debug (prompt-cache hit)** | **~1.5k effective** | **~900-1300** | **~2-3s** | **~$0.005** |

Cost is roughly 2-3× a normal call. Trivial in absolute terms, but the
extra is opt-in only.

## 5. Testing

### 5.1 Updated tests

`tests/test_agent.py`:

- All existing tests that destructure `AgentResult` continue to work
  because the new fields have defaults. They are NOT touched.
- `test_pick_returns_validated_ids` adds an assertion that
  `result.raw_picks == [1, 3]` and `result.pick_reasons == {}`.
- `test_pick_drops_invalid_ids` adds an assertion that
  `result.raw_picks == [1, 99, 3]` (includes the dropped id) and
  `result.picks == [1, 3]`.

### 5.2 New tests

`tests/test_agent.py`:

- `test_system_prompt_includes_count_override_instruction` — asserts
  the literal text *"If the user's query specifies a number of
  tracks"* is in `SYSTEM_PROMPT`.
- `test_user_blocks_say_default_count_not_desired_count` — asserts
  the second user block contains `"Default count:"` not `"Desired
  count:"`.
- `test_pick_in_debug_mode_appends_suffix_to_system_prompt` — patches
  `llm.call`, verifies the `system` kwarg contains `DEBUG_SUFFIX`'s
  literal substring `"DIAGNOSTIC MODE"`.
- `test_pick_in_debug_mode_parses_pick_reasons` — mocks the LLM to
  return JSON with `pick_reasons`, asserts the resulting `AgentResult`
  carries the parsed dict.
- `test_pick_in_normal_mode_ignores_pick_reasons_field` — even if
  the mocked LLM returns `pick_reasons`, the result has an empty dict
  (we don't accidentally surface it when `debug=False`).

`tests/test_cli.py`:

- `test_debug_flag_prints_sections_to_stderr` — captures stderr via
  `capsys.readouterr().err`, asserts the literal strings
  `=== config ===`, `=== library context ===`, `=== query ===`,
  `=== prompt ===`, `=== haiku call ===`, `=== haiku response ===`,
  `=== validation ===`, `=== output ===` are all present, in order.
- `test_debug_flag_keeps_stdout_clean_for_alfred_pipelining` — runs
  with `--debug --alfred-json`, asserts that **stdout** is parseable
  JSON (no debug text mixed in).

### 5.3 No e2e test changes

`tests/test_e2e.py` continues to call `agent.pick(..., debug=False)`
and is unaffected.

## 6. Error handling

- If `--debug` is set and Haiku returns malformed `pick_reasons` (e.g.
  a list instead of a dict), `_extract_json` already raises
  `MalformedJSONError`. Caller (CLI) catches this the same way as any
  parse failure. No new error path.
- If `--debug` is set but `pick_reasons` is missing from the response,
  treat it as `{}` and proceed. The CLI diagnostic section just shows
  "(no per-track reasons returned)".
- Stderr-print failures (closed pipe) are ignored — we wrap the
  debug-print calls in `try/except OSError: pass`.

## 7. Migration

None. The two new `AgentResult` fields default to empty values, so
existing code that constructs an `AgentResult` (only `agent.pick` does
that) keeps working without changes. The cache schema is unchanged.
Old playlists remain valid.

## 10. Decision log

- **2026-05-14** — `--debug` is a single flag; no separate
  `moodlist explain` subcommand.
- **2026-05-14** — debug output goes to stderr; stdout stays clean
  for Alfred / piping compatibility.
- **2026-05-14** — count from query is handled by Haiku via prompt
  instruction, not by regex. Trusts the LLM to read English.
- **2026-05-14** — per-track reasons are opt-in only (debug mode),
  to keep normal-mode token cost unchanged.
- **2026-05-14** — `AgentResult` grows two new fields with defaults;
  no breaking changes to existing call sites.
